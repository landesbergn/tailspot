Objective. Make the Collection economy clear, accurate, consistent, and scalable, and make the catch land as a moment worth returning for — without changing the catch engine.

Product Contract

Summary

Re-tier every aircraft to the research-grounded list and re-balance points to a flat 5-tier curve (10/20/50/100/500) with a first-of-type bonus; collapse the three rarity resolvers and two point ladders into one tested source; re-score existing catches. Pass adsb.lol route data through to the catch. Replace the reveal with the minimal + split-flap card carrying the photo, altitude/speed/route, and a score ledger, plus an occasional route-guess bonus round.

Problem Frame

Today the economy is opaque and inconsistent: flat-per-tier points are arbitrary; the rarity→points ladder is hand-duplicated in iOS and the backend with no shared test; a third string-classifier ladder and hand-set Sets.swift rarities can drift; and tier assignment is coarse (every widebody “uncommon”, all military “rare”, warbirds and vanishing airliners lumped into “common”). The reveal is a flat card that doesn’t make a rare catch feel rare, and the catch carries no route. The result understates real catches (the 940→2755 leaderboard bug) and leaves the collection loop without a payoff moment.

Acceptance Examples

  • AE1 (re-tier). A caught B744 resolves to rare (50) on both app and backend; a B38M to common (10); an A388 to rare; a B721 to legendary.
  • AE2 (first-of-type). The device’s first B744 ever scores 50 + 25 = 75; a later B744 scores 50.
  • AE3 (guess). On a routed catch in a guess round, picking the correct far airport adds round(points × 0.10); a wrong pick or skip adds nothing; the route is not shown until after the pick.
  • AE4 (no route). A catch whose flight has no adsb.lol route shows no route line and never triggers a guess round; everything else works.
  • AE5 (parity). Editing the iOS point table without the backend (or vice-versa) fails the parity test in CI.

Planning Contract

Key Technical Decisions

KTD1 Curated tiering, research-grounded

Tiers stay a hand-authored table in the generator (not a data pipeline) — re-tiered by judgment to the research list. Frequency is a tool consulted while authoring, not a runtime system. (D1 Fork 1)

KTD2 Flat points + bonuses

Flat per tier 10/20/50/100/500; first-of-type +50% of base; route guess +10% of catch points. No bands, no decay. (D1 Forks 2–3, D2)

KTD3 One source of truth + parity test

Collapse the three rarity resolvers and two point ladders into one source both platforms read; a CI parity test fails on drift. The string-classifier fallback stops carrying its own tier ladder (R6). (D1 “one code path”)

KTD4 Base live, bonuses frozen

Base points re-derive live from resolvedRarity (so re-tiers correct old catches on read); per-catch bonuses (first-of-type flag, guess result/bonus) freeze on the Catch. The reveal ledger = live base + frozen bonuses. The server’s scored total is authoritative for the leaderboard; iOS reconstructs the ledger for display.

KTD5 First-of-type scored server-side

The canonical scoreCatch gains device context and checks the device’s catch history for the typecode at upload — authoritative, un-spoofable. iOS may show an optimistic chip, reconciled to the server value.

KTD6 Route via adsb.lol passthrough

Add route (origin/dest airports) to NormalizedAircraft → /v1/aircraft → iOS Aircraft → Catch, additive — the exact t/r/category pattern. Execution-time confirm: whether adsb.lol carries route inline in the aircraft feed or needs its routeset endpoint (a per-callsign lookup + cache).

KTD7 Re-score discipline

Bump CURRENT_SCORING_VERSION; rescore --dry-run prints the before→after delta + tier transitions; Noah reviews before the prod run. Mirrors the 940→2755 fix. Migration applied to prod manually (no release_command).

KTD8 Reveal is a new SwiftUI surface

Rework CardReveal into the minimal + split-flap card (photo, data, ledger), intensity scaling by tier; seed from the design prototype. Photo is the v1 medium. The front ledger carries the breakdown, so a separate flip-to-receipt is dropped (the flip, if kept, is airframe detail only).

KTD9 The guess round

A pure helper picks the endpoint farther from the observer and generates 4 options (real airport + plausible same-region decoys); exact-airport correctness; cadence every 3–5 catches with randomness, gated on a known route; +10% computed client-side and confirmed/owned by the server.

High-Level Design

Assumptions & sequencing

  • Phase 1 (economy + data, U1–U8) lands first and is independently shippable; Phase 2 (reveal + guess, U9–U12) builds on it.
  • adsb.lol exposes usable route data (the premise of #6); the exact mechanism is confirmed in U6.
  • SwiftData migration stays additive (new optional fields, defaults) — testers keep their catches.
  • The point values move the leaderboard; the dry-run re-score is the review gate before any prod change.

U11 Route-guess bonus round Phase 2

Goal
An occasional pre-reveal guess of the far endpoint for +10%.
Reqs
R4 · R12
Files
ContentView.swift (catch flow + cadence), new GuessStep view, a pure guess helper, Catch.swift (frozen guess fields), scoring
APPROACH
  • Pure helper: given observer location + route endpoints, pick the farther endpoint (the one to guess) and the prompt (“coming from” vs “headed to”); generate 4 options (real airport + 3 plausible same-region decoys).
  • Cadence: trigger every 3–5 catches with randomness, only when the route is known. Route hidden during the guess; resolved on the reveal.
  • Correctness = exact airport. On correct, add round(catchPoints × 0.10); persist guessedAirport + result + bonus frozen on the Catch. Client computes; the server owns the authoritative total (validate on upload).
TEST SCENARIOS
  • Covers AE3, AE4. Far-endpoint pick: observer near SFO + HKG→SFO → asks origin (HKG); observer near HKG → asks destination. Option set always contains the answer + 3 unique decoys.
  • Cadence helper fires within the 3–5 window and never on a routeless catch. Bonus = 10% rounded; wrong/skip → 0. Guess fields freeze on the Catch.
VERIFICATION

Helper unit tests green; the round triggers and resolves correctly on device.
