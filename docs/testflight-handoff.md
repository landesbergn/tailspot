# TestFlight v0 handoff checklist

What's already done (committed to the repo) and what Noah needs to do
in Apple's UIs to get the first TestFlight build into a tester's hands.

## What's already done

1. **Credential plumbing for shipped builds.** OpenSky OAuth credentials
   read from `Bundle.main.infoDictionary` at runtime, baked there at
   build time from `ios/Tailspot/Tailspot.secrets.xcconfig` (gitignored).
   Env-var fallback preserves the dev iteration loop. See "Credentials
   setup" below for the one-time local config.

2. **Manual `Info.plist`** at `ios/Tailspot/Info.plist`. Replaces the
   auto-generated one. Adds `ITSAppUsesNonExemptEncryption=false`
   (skips export compliance questionnaire), and the custom
   `OpenSkyClientID` / `OpenSkyClientSecret` keys.

3. **Privacy manifest** at
   `ios/Tailspot/Tailspot/PrivacyInfo.xcprivacy`. Declares
   UserDefaults + FileTimestamp required-reason API usage, and the
   precise-location + photos data categories (app-functionality only,
   not linked, not tracking).

4. **Version + build number.** `MARKETING_VERSION=0.1.0`,
   `CURRENT_PROJECT_VERSION=1`. Bump scheme: increment the build
   number on every TestFlight upload (App Store Connect requires
   monotonically increasing values per marketing version).

5. **App icon.** 3 luminosity variants (light / dark / tinted) at
   1024×1024 in `Assets.xcassets/AppIcon.appiconset/`. Generated by
   `tools/generate-app-icon.swift` — re-run if you tweak the script,
   or replace the PNGs directly with a designed icon.

## Credentials setup (one-time, on your Mac)

1. Copy the template:
   ```
   cp ios/Tailspot/Tailspot.secrets.example.xcconfig ios/Tailspot/Tailspot.secrets.xcconfig
   ```
2. Open `Tailspot.secrets.xcconfig` and paste your OpenSky client_id +
   client_secret. The file is gitignored — verify it's NOT shown in
   `git status` before committing anything.
3. Clean build (in Xcode: `Product → Clean Build Folder`, or
   `xcodebuild clean ...`). Xcode caches resolved Info.plist values
   per-build; the clean ensures the next build picks up the new values.

After this, the credentials flow into the shipped binary. They're
extractable by anyone with the `.ipa`; this is the accepted v0 risk.
Rotate the OpenSky secret if/when you stop sharing with the trusted
testers.

## What you need to do in Apple's UIs

### 1. App Store Connect — create the app record

You only do this once per app. https://appstoreconnect.apple.com →
My Apps → "+" → New App.

- **Platforms**: iOS
- **Name**: Tailspot
- **Primary language**: English (US)
- **Bundle ID**: select `com.landesberg.Tailspot` (Apple Developer
  account → Identifiers; if it doesn't appear in the dropdown,
  register the ID there first — Identifiers → "+" → App IDs → App
  → Description "Tailspot", Bundle ID "Explicit" with value
  `com.landesberg.Tailspot`).
- **SKU**: anything unique, e.g. `tailspot-ios-v0`.
- **User Access**: Full Access (just you).

You'll need to also set the App Information → Category (Lifestyle?
Travel? Pick what fits — can change later). Privacy Policy URL is
required for TestFlight external testers but NOT for internal (your
own Apple ID + up to 100 internal testers — sufficient for v0). For
external (up to 10,000 testers), you'll need a privacy policy URL
later.

### 2. Signing — let Xcode manage it

In Xcode → open `ios/Tailspot/Tailspot.xcodeproj` → select the
Tailspot project (top of left sidebar) → Tailspot target →
Signing & Capabilities tab.

- **Automatically manage signing**: checked
- **Team**: select your Apple Developer team (G9FJX2A5TA, already
  set in project.pbxproj)
- **Bundle Identifier**: `com.landesberg.Tailspot` (already set)

Xcode will create + download the necessary signing identity +
provisioning profile in the background. If it complains about a
missing certificate, click "Try Again" — Xcode will offer to fix.

### 3. Archive + Upload

In Xcode:

1. Change the run destination at the top of the window from "iPhone
   17 Simulator" to **Any iOS Device (arm64)**. You can't archive for
   a simulator.
2. `Product → Archive`. This takes a few minutes — it builds a
   release configuration, signs with your distribution certificate,
   and bundles into an `.xcarchive`.
3. When the archive finishes, the **Organizer** window opens
   (`Window → Organizer` if it doesn't).
4. Select the new archive → **Distribute App**.
5. Choose **App Store Connect** → **Upload**.
6. Accept the defaults on the next screens (Automatically manage
   signing, Strip Swift symbols = Yes, Upload your app's symbols = Yes).
7. **Click Upload.** The upload takes a few minutes.

If the upload fails citing "missing compliance" or an export-related
prompt — the `ITSAppUsesNonExemptEncryption=false` key in our
Info.plist should suppress that. If it still asks, answer "No, my app
does not use non-exempt encryption."

### 4. Wait for processing

After the upload, Apple processes the binary — usually 5–20 minutes.
You can watch progress in App Store Connect → My Apps → Tailspot
→ TestFlight → Builds. The build first appears as "Processing", then
"Ready to Submit" (no review needed for internal testing).

### 5. Add internal testers

App Store Connect → TestFlight → Internal Testing → "+" → Create
group, e.g. "Closed alpha". Add yourself + your 1–2 testers by Apple
ID email. They'll get an invite email; clicking the link prompts them
to install TestFlight on their iPhone, then accept the invite, then
install Tailspot.

**Internal testers** must be members of your Apple Developer team —
you add their Apple ID under Users and Access → People before they
can be added here. (For external testers, you skip the team add but
the build needs Apple's beta review, which takes 24-48 hours.)

For 1–2 trusted people, internal is the easier path.

### 6. (Optional) Add release notes

In the same TestFlight Builds view → click the build → "Test Details"
→ "What to Test" field. Tell testers what to try and what's broken.
Suggested seed for v0:

> Tailspot v0.1.0 (build 1). Field-tested in Berkeley / Oakland.
>
> **Try:** Step outside and point your phone at the sky. Planes should
> get a cyan corner-bracket label with callsign + airline + model.
> Tap a plane to pin its lock; tap the capture button to "catch" it.
> Open the Hangar (top-right tray icon) to see your collection
> organized by set or by recent activity.
>
> **Known limits:** OpenSky doesn't cover most small GA, helicopters,
> or military traffic. Indoor tests will show "no aircraft in view."
> Compass calibration matters — tap the compass icon to recalibrate.
>
> **Feedback** welcome to noah.landesberg@gmail.com — what works,
> what's broken, what's confusing.

## Xcode Cloud (optional CI path)

If you're using Xcode Cloud instead of (or in addition to) the manual
Archive flow above, there are two settings to get right that aren't
obvious from the default workflow setup.

### 1. Workflow project path

The repo's Xcode project sits at `ios/Tailspot/Tailspot.xcodeproj`,
not at the repo root. Xcode Cloud's default workflow scans the root,
finds nothing, and fails with the misleading error:

> A scheme called Tailspot (Tailspot project) does not exist in
> Tailspot.xcodeproj

Fix: in Xcode → Cmd+9 (Report Navigator) → Cloud tab → right-click
the workflow → Edit Workflow → on the build/archive action set
**Project or Workspace** to `ios/Tailspot/Tailspot.xcodeproj`. Or in
App Store Connect → Xcode Cloud → Manage Workflows → same fields.

### 2. OpenSky credentials in CI

`Tailspot.secrets.xcconfig` is gitignored, so Xcode Cloud's clone
won't have it — the resulting build runs in anonymous OpenSky mode
(400 credits/day, exhausted in ~1.3hr per IP). To fix:

1. In your workflow → Environment → Environment Variables →
   add two **secret** entries:
   - `OPENSKY_CLIENT_ID`
   - `OPENSKY_CLIENT_SECRET`
2. The committed `ios/Tailspot/ci_scripts/ci_post_clone.sh` reads
   those env vars and writes them to
   `ios/Tailspot/Tailspot.secrets.xcconfig` before the build phase
   runs. Xcode Cloud invokes this automatically — no further config
   needed on your side.
3. The build log will print a line like
   `ci_post_clone: wrote .../Tailspot.secrets.xcconfig (lengths: id=8, secret=64)`
   so you can confirm from the log that the script saw real strings.
   Values are never logged.

If the env vars are missing or empty, the script logs that it's
skipping and the build still succeeds (anonymous mode). Fix when
ready.

## Subsequent uploads

Each new TestFlight build needs:

1. `CURRENT_PROJECT_VERSION` bumped (1 → 2 → 3...) in BOTH the Debug
   and Release configs of the Tailspot target in `project.pbxproj`.
2. (When shipping a meaningful feature set) bump `MARKETING_VERSION`
   too: `0.1.0` → `0.1.1` etc.
3. Repeat steps 3–6 of the upload flow.

Testers running the previous build will see the new one as an update
in TestFlight on their device.

## If things go wrong

- **"No accounts with iTunes Connect access have been found"** at
  upload time: open Xcode → Settings → Accounts, ensure your Apple
  ID is signed in and has Admin or App Manager role in App Store
  Connect.
- **"Invalid Provisioning Profile"** during archive: in
  Signing & Capabilities, toggle automatically-manage-signing off
  and back on; this usually re-pulls a fresh profile.
- **OpenSky returns 401 in shipped builds**: the credentials in
  `Tailspot.secrets.xcconfig` aren't flowing through. Verify via
  `plutil -p /path/to/Tailspot.app/Info.plist | grep OpenSky` after
  the archive — the values should be non-empty. If they're empty,
  clean build + reupload.
- **"This bundle is invalid. The Info.plist file is missing the
  required key…"** — could be `LSRequiresIPhoneOS`, `UIRequiredDevice
  Capabilities`, or similar. Compare our Info.plist against a known-
  good one; add the missing key.

## Backstop: rotate creds if a tester's device gets compromised

The OpenSky `client_secret` is in the shipped binary. If a tester's
phone is lost, stolen, or jailbroken, treat the secret as leaked and
rotate via OpenSky's API console
(https://opensky-network.org/my-opensky/api-credentials), then:
1. Update `Tailspot.secrets.xcconfig` with the new values.
2. Clean build + upload a new build.
3. Old TestFlight build continues to use the old (rotated) secret
   until the tester updates — which is fine because it's now invalid.

For wider distribution, the backend proxy (PLAN.md §1) replaces this
flow entirely; the client never holds the secret.
