Decision 1 — The Scoring Model

How a plane’s rarity becomes points. Decisions 2 (the reveal) and 3 (trophies) both pay out of this, so it goes first. The whole model: a plane has a rarity tier; the tier is worth a fixed number of points. Everything below is in service of keeping it that simple.

Principles

Three, in priority order. They decide every fork below.

1 · Simple

The model fits in one sentence: tier → points. No bands, no decay, no per-region math. If a rule needs a diagram to explain, it’s too complex for this game.

2 · One source of truth, tested

Tier and points live in one place, shared by the app and the backend, covered by tests. Adding a plane is one edit. The framework grows without rework.

3 · Rarity feels earned

Points map straight to the tier, so a rare catch is plainly a bigger prize than a common — and the player can see why. Rare should feel rare.

These map to the strategy: simplicity and a clean code path (your steer), and a collection game where rarity feels earned (Bet B — catching stays compulsive; trust is the north-star). Not principles: anti-grind and forcing variety — catching a lot of one plane is fine; it may be all that’s overhead.

What’s wrong today

Three real problems. Note what is not here: flat points-per-tier is fine, and there’s no grind to fix.

#ProblemEffect
1Three rarity paths, two copied point ladders. Tier resolves three ways (live AR, Catch.resolvedRarity, backend); point values are separate literals in GameSystem.swift and backend/catches/points.ts.Edit one, forget another → app total ≠ leaderboard total. Drift, untested.
2Sets carry hand-set rarities. Sets.swift nil-typecode entries aren’t pinned to the table.The one real divergence left after the June fixes.
3Tier assignment is coarse and values are arbitrary. e.g. every widebody → uncommon, regardless of how often it actually flies.Rarity doesn’t always feel right — a daily A330 and a monthly 777F are lumped together.

Already good (keep): one tier table feeds both platforms (AircraftTypes.json), Catch.resolvedRarity re-derives live so re-tiers fix old catches on read, and the versioned rescore job re-prices the board on command.

Fork 1 — where does a tier come from?

One hand-authored table, or grounded in real flight frequency? Under the simplicity principle, the answer flips from my earlier draft.

Curated table recommend

One table, 5 tiers, authored by judgment. Adding a plane is one row. Simplest, one tested path, trivial to extend. You can look at real frequency when deciding a tier — but the system stays a table.

Hybrid deferred

A frequency signal nudges tiers automatically. More accurate, but a second moving part to build and keep running. Against principle 1. The framework can add it later with no rework (principle 2) — so not now.

Data-derived no

Tiers computed from frequency percentiles. Most accurate, least simple, needs a live pipeline and regional baselines. Wrong fit for this game today.

Fork 1 Locked: curated; tiers grounded in research. It’s the simplest thing that fixes problem 3: re-tier the coarse buckets by hand (you decide a 777F outranks an A330), all in the one generator that feeds both platforms. Frequency stays a tool you consult, not a system to maintain. Door open to automate later. The current tier rules + override list are laid out for your review in the companion Tiering Review doc.

Fork 2 — tiers and points

Keep 5 tiers and one flat point value each — the simplest shape, and it’s what ships today. Two settled, one open:

  • 5 tiers — Common · Uncommon · Rare · Epic · Legendary. Five is the legible ceiling; one-of-a-kind airframes (An-225, Air Force One) sit at the top of Legendary, not in a 6th tier.
  • Flat per tier — every Rare is worth the same. Simple and predictable. (No bands.)
  • The five numbers — the one open call, and it’s taste.

Today’s values are 10 / 25 / 100 / 500 / 2000 — a 50× jump from Common to Epic. You want that flatter: gentle through the middle, the very rarest still worth a lot. Proposed: 10 / 20 / 50 / 100 / 500. Common→Epic drops from 50× to 10×; Legendary keeps a deliberate 5× jump over Epic, so the grail still towers.

Fork 2 Locked: 5 tiers, flat values, 10 / 20 / 50 / 100 / 500. Legendary is the easy dial — say the word to push the grail higher (750–1000) or lower.

Fork 3 — one optional bonus: first-of-type

The core is just tier → points. There’s exactly one bonus worth considering, and it earns its complexity only if you want it: a one-time bonus the first time you ever catch a type.

Add first-of-type

First catch of a new type pays extra (say +50%). Rewards collecting — the point of Bet B — without penalizing anyone for catching many of one plane. One rule, easy to build (we already know your caught types).

Skip it — tier → points only

The strictest reading of “simple”. A new rare already pays more than a common; the Hangar already marks new types. No extra rule in the economy at all.

Fork 3 Locked: add it — first catch of a type pays +50%. The one bonus that directly serves the collection game, and it stays one sentence.

Fork 4 — global or regional rarity?

An A320 is common worldwide. Should it be worth less in a city where it’s everywhere, and more somewhere it’s rare?

Global recommend

One tier per plane, everywhere. Simple and legible — the same catch is always worth the same. Where you are doesn’t silently change the math.

Regional no

Locally-rare planes pay more locally. “Fairer” for a quiet region, but the same A320 now scores differently city to city, and it needs per-region data. Against principles 1 and 2.

Fork 4 Locked: global.

One code path (principle 2, made concrete)

This is the part you flagged as load-bearing: internal consistency, one clear path, tests. The fix for problem 1 — collapse the three rarity resolvers and two point ladders into a single source the app and backend both read, with a parity test that fails if they drift.

Rolling out any change stays the proven flow: edit the table/values → bump scoring_version → rescore --dry-run → review the delta → run it. That’s what makes the framework safe to grow.
